# Clinical Patient Profile HTML Skill

## 这是什么

这是一个用于自动制作临床项目 `patient profile` 的 Skill。

它的目标是把临床数据团队、医学团队、项目管理团队在核查准备、医学审阅、中心审阅时常用的 `patient profile` 制作流程固化下来。  
使用者不需要手工整理图表或逐个受试者拼接信息，只需要上传项目文件并在对话中提出需求，Skill 就会自动检查文件、识别数据、并生成交互式 HTML。

生成结果以中文为主，呈现风格与既往项目中使用的 `patient_profile_two_centers.html` 保持基本一致。

其中，疗效指标不再默认沿用某个既往项目的固定口径，而是优先依据研究方案中的：

- 主要终点
- 关键次要终点 / 次要终点
- 研究流程表 / 评估时间表

来决定：

- 本项目应该纳入哪些疗效指标
- 这些指标是连续型变量还是二分类变量
- 是否需要展示关键应答判定

## 适合谁用

- 非技术人员
- 临床项目经理
- 医学经理
- 临床运营
- QA
- 注册同事
- 统计同事

## 能做什么

Skill 会基于上传的数据文件，自动生成：

- 受试者层面的 patient profile
- 中心层面的汇总 profile
- 疗效历时变化
- 实验室检查历时变化
- 生命体征历时变化
- Finding 固定展示模块
- 关联风险和现场核查回复信息
- 配套的清洗后数据表和质控输出

## 怎么触发

在对话中出现以下任一类表达时，Skill 应自动启动：

- `制作patient profile`
- `patient profile`
- `制作受试者画像`
- `制作个例profile`
- `制作中心汇总profile`

如果用户已经上传了文件，并且只说一句 `制作patient profile`，Skill 应直接进入执行流程：

1. 先做文件预检查
2. 先确认两件范围问题：
   - 纳入全部受试者，还是仅纳入已随机受试者
   - 是否纳入 USV / 计划外访视数据
3. 如果只有提示项，则直接开始构建
4. 如果有阻断项或关键映射不清楚，再最小化追问
5. 如果用户确认某些字段或某类数据本轮不纳入，则按缩减范围直接继续构建

不需要用户再额外说一次“开始构建”。

## 需要上传什么文件

### 必需文件

1. 主 data listing
   - 受试者层级 listing
   - 通常应包含人口统计学、随机、访视、疗效、实验室、AE、生命体征等 sheet

2. Finding 文件
   - 自查问题汇总
   - 稽查问题汇总
   - 或其他受试者级 Finding / issue tracker

3. 研究方案 / 勘误 / 修订说明
   - 这是疗效识别的核心输入
   - Skill 会从方案中解构主次要终点和研究流程，再决定疗效部分怎么做

### 建议上传文件

4. 受试者报表
   - 用于交叉核对性别、年龄组、随机组别等关键信息

5. 已有参考 HTML
   - 如果项目希望延续既往展示风格，可作为只读参考

## Skill 会先检查什么

Skill 启动后，会先做一轮预检查，判断：

- 主 listing 是否存在
- Finding 文件是否存在
- 研究方案是否存在、是否可读取
- 方案中是否能识别主要/次要终点
- 方案中是否能识别研究流程 / 疗效评估安排
- 是否识别到中心
- 是否识别到关键 sheet
- 疗效 sheet 是否能与方案中的终点指标对应
- 实验室 sheet 是否可识别
- Finding sheet 和受试者字段是否能定位
- 关键结果列、日期列是否能识别
- 疗效指标是连续型还是二分类是否能判断

预检查会输出：

- `input_precheck.md`
- `input_precheck.json`
- `suggested_project_config.json`
- `protocol_endpoint_summary.md`
- `protocol_endpoint_summary.json`

## 什么情况下会追问用户

只有在下面这些情况，Skill 才应该追问：

- 主 listing 缺失
- Finding 文件缺失
- 研究方案缺失
- 方案无法读取
- 方案中不能明确定位主次要终点
- 方案中不能判断某疗效指标是连续型还是二分类
- 尚未确认纳入全部受试者，还是仅纳入已随机受试者
- 尚未确认是否纳入 USV / 计划外访视数据
- 中心范围不明确
- Finding sheet 对应中心或受试者字段无法确认
- 关键疗效或实验室 sheet 无法识别
- 研究流程表显示某类检查需要收集，但 listing 中未找到对应数据
- 某个关键指标的结果列或日期列不明确
- 筛选日期、知情同意日期、基线/随机日期等关键字段无法稳定定位
- 用户需要决定某类字段是否不纳入本轮 profile

追问原则是：

- 只问最小必要问题
- 不重复确认已经明确的信息
- 不要求用户再说“开始构建”

## 用户需要确认的典型问题

例如：

- 本轮纳入全部受试者，还是只纳入已随机受试者
- 是否纳入 USV / 计划外访视数据
- 本轮纳入全部中心，还是只纳入某几个中心
- 某个疗效指标虽然在 listing 中存在，但方案中并不是本项目终点，是否不纳入
- 某个终点在方案中提到，但 listing 中没有可对应的 sheet，是否先排除
- 某个疗效指标应按连续型展示还是按二分类展示
- 某个字段是否缺失但允许不纳入
- 某个 Finding sheet 是否暂不纳入
- 某类实验室或生命体征是否无需展示

如果用户明确说“不需要体现某些数据”，Skill 应视为范围确认，继续构建。

## HTML 大致长什么样

生成的 HTML 是离线可打开的中文交互式页面，通常包含：

### 顶部筛选区

- 中心
- 受试者
- 疗效指标
- 实验室指标
- 生命体征指标
- 异常值筛选
- Finding 筛选
- 关键词搜索

### 受试者层

- 受试者基本信息卡片
- 基于研究流程表自动识别的研究分期进入/完成日期
- 疗效模块
- 实验室检查模块
- 生命体征模块
- Finding 固定展示模块

### 中心层

- 中心 flowchart / profile 汇总
- 按试验组 / 对照组拆分的汇总表格
- 连续型变量的均值 ± 标准差历时图
- 二分类变量的应答比例图

### 附录和支持信息

- 字段映射附录
- 方案终点解构摘要
- 质控输出
- 未解决问题清单

## 输出风格

HTML 默认：

- 中文标题
- 中文标签
- 中文图表说明
- 中文表格列名
- 中文 Finding 回复建议

整体视觉风格保持专业、简洁、适合药企内部审阅、联合核查准备会和现场核查使用。

## 对数据准确性的要求

Skill 的基本原则是：

- 不编造数据
- 不把前一个项目的疗效指标口径直接复用到新项目
- 以研究方案中的主次要终点和研究流程作为疗效识别优先依据
- 以研究流程表识别研究分期，而不是默认套用“双盲期”“开放期”等既往项目标签
- 不反推不存在的访视日期
- 不自行补正常值范围
- 不把缺失值当作 0
- 不删除异常值
- 定性指标不强行数值化
- 以 source listing 为准
- 如果字段映射不确定，或关键字段无法稳定识别，就先停下来提示用户确认，而不是静默猜测

## 对非技术同事的建议用法

最简单的使用方式是：

1. 把项目文件放进同一个文件夹
2. 在对话里上传这些文件
3. 直接说一句：`制作patient profile`
4. 如果 Skill 提问，就按提示确认中心、字段或不纳入范围
5. 等待 HTML 和配套文件输出

## 备注

这个 Skill 既支持：

- 用默认预检查自动识别文件、方案终点和字段映射

也支持：

- 在预检查后人工调整 `suggested_project_config.json`

因此既适合标准化项目，也适合结构相近但不完全相同的新项目。
